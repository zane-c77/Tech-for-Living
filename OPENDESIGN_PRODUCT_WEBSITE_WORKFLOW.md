# OpenDesign 产品网站设计流程说明

面向对象：老板 / CTO  
作者：Zane  
日期：2026-07-13

## 一句话总结

我的方法不是单纯让 OpenDesign 自动生成网站，而是把 OpenDesign 放进一套可复制的产品网站设计流程里：先用模板确定网站结构，再用 LLM 整理产品资料和生成 prompt，然后用 OpenDesign 快速出 first draft，最后通过人工判断和 AI 检查反复迭代，尽快得到一个可以给老板评审的实用版本。

## 我们做的网站类型

目前主要做的是产品网站。它的目标不是单纯展示视觉效果，而是清楚介绍一个产品是什么、给谁用、解决什么问题、有哪些功能、价格如何、怎么联系或转化。

所以我更关注以下几件事：

- 网站结构是否清楚
- 产品卖点是否突出
- Hero 首屏是否能吸引人
- 功能、用途、价格、联系方式是否完整
- 文案是否准确、通顺、适合目标用户
- 页面布局、字体、动效是否符合产品调性
- 是否能快速得到一个可讨论、可修改的 first draft

## 工具分工

| 角色 | 主要作用 |
| --- | --- |
| OpenDesign | 快速生成产品网站初版，完成页面布局、视觉风格、动效和页面呈现 |
| ChatGPT / LLM | 总结产品资料、整理网站内容、生成适合 OpenDesign 的 prompt、检查逻辑和文案 |
| 人工判断 | 判断文案准确性、页面结构合理性、产品重点是否突出、是否符合老板预期 |

## 标准流程

### 1. 先套一个产品网站模板

第一步我会先套一个网站模板。这个模板不一定是最终视觉设计，而是作为网站的结构骨架。

常见模块包括：

- Hero 首屏
- 产品介绍
- 目标用户
- 使用场景
- 核心功能
- 价格或方案
- 联系方式
- FAQ 或补充说明

这样做的目的是让后续添加内容和修改内容更有结构，而不是随机堆页面。产品网站最重要的是信息层级清楚，所以我会先用模板保证网站有一个稳定的框架。

### 2. 用 LLM 总结产品资料

第二步，我会把产品相关资料交给 ChatGPT 或其他 LLM，让它帮我整理成结构清晰的网站内容。

我通常会让 LLM 总结出：

- 目标用户
- 产品介绍
- 产品用途
- 使用场景
- 主要功能
- 价格信息
- 联系方式
- 可以放到页面里的核心卖点

这一步的价值是把原本零散的产品资料整理成适合网站使用的内容模块。整理完成后，我就有了一份比较完整的网站内容基础。

### 3. 根据模板和内容生成 OpenDesign prompt

第三步，我会结合第一步的网站模板和第二步整理出的产品内容，让 ChatGPT 帮我写适合 OpenDesign 使用的 prompt。

这个 prompt 通常会包含：

- 产品是什么
- 面向什么用户
- 网站应该有什么页面模块
- 整体设计调性
- 参考的页面布局
- 字体和视觉风格
- 交互动效要求
- 价格展示方式
- 文案和内容重点

我一般会先用自然语言把需求描述给 ChatGPT，再让它改写成更适合 OpenDesign 的 prompt。如果 ChatGPT 写得太长，我会让它压缩；如果内容已经清楚，大部分时候我会直接拿去 OpenDesign 使用。

### 4. 参考同类网站的可复制部分

在生成网站之前，我会让 LLM 或自己参考一些符合产品调性的网站。

重点参考的是容易复制和落地的部分：

- 页面布局
- 字体风格
- 按钮样式
- 交互动效
- 内容展示方式
- 价格卡片设计
- 产品卖点的表达方式

我不是直接照抄某一个网站，而是参考它的信息组织方式和视觉表达方式，再结合我们的产品内容生成一个最快的实用版本。

### 5. 用 OpenDesign 生成 first draft

有了模板结构、产品内容和 prompt 后，我会让 OpenDesign 生成网站初版。

这个阶段的目标不是一次做到完美，而是快速得到一个可以看的 first draft。这样老板、客户或团队可以更早看到具体页面，而不是停留在抽象讨论。

### 6. 检查初版质量

OpenDesign 生成初版后，我会重点检查几个地方。

首先是 Hero 首屏。Hero 是用户第一眼看到的地方，必须足够吸引人，也要清楚表达产品价值。如果 Hero 不满意，我通常会继续找几个方向重新生成或修改。

然后我会检查：

- 功能介绍有没有错误
- 产品资料有没有遗漏
- 文案是否通顺
- 页面顺序是否合理
- 价格和联系方式是否清楚
- 文字设计是否符合网站调性
- 动效设计有没有做到 prompt 里的要求

### 7. AI 检查加人工判断

我一般会先自己完整看一遍网站，判断它是否通顺、重点是否清楚、布局是否合理。

然后我会再给 AI 看一遍，让它检查：

- 内容逻辑是否顺
- 文案是否重复或空泛
- 产品介绍是否缺失
- 页面结构是否符合产品网站
- 有没有明显不符合目标用户的表达

如果整体通顺，并且没有明显错误，我会先给老板看 first draft。

### 8. 在 OpenDesign 中迭代修改

我在 OpenDesign 里主要修改的是：

- 文案
- 页面顺序
- 模块删减
- 局部内容调整

视觉和动效方面，我更多是通过 LLM 生成新的 prompt，再发给 OpenDesign 修改。如果不满意，我会把问题反馈给 LLM，让它重新写 prompt，然后继续重复这个过程。

## 人工判断和 AI 的边界

我会依赖 AI 完成这些部分：

- 产品资料总结
- 网站结构整理
- OpenDesign prompt 生成
- 初版页面生成
- 文案通顺度检查
- 动效描述和修改建议

但以下部分必须人工判断：

- 产品重点是否准确
- 文案是否符合真实业务
- 页面布局是否合理
- Hero 是否足够吸引人
- 网站是否符合老板或客户预期
- OpenDesign 是否真的完成了修改

也就是说，AI 和 OpenDesign 可以大幅提高出稿速度，但最终的网站质量仍然需要人来判断和把关。

## OpenDesign 的最大价值

OpenDesign 最大的价值是快速出稿。

它能让我在较短时间内，把一个产品网站从想法变成可视化页面。这样可以更快进入讨论和修改阶段，而不是一开始就花大量时间在空白页面上设计。

对团队来说，它的价值主要是：

- 缩短 first draft 时间
- 降低产品网站设计门槛
- 更快对齐老板或客户预期
- 让设计讨论从抽象变成具体页面
- 方便基于反馈持续迭代

## 当前遇到的问题

我最常遇到的问题是：OpenDesign 有时会反馈说已经修改完成，但实际检查页面时发现并没有真正改到位。

即使 prompt 里已经要求它检查，也还是可能发生。所以我目前的处理方式是：

- 每次修改后都人工检查
- 如果没改到位，就重复告诉它具体问题
- 必要时让 ChatGPT 重新生成更明确的 prompt
- 不完全相信工具的完成反馈，以实际页面结果为准

这个问题也说明，如果要把流程做成 skill，必须加入明确的检查步骤，而不是只生成 prompt。

## 如果做成 Skill，建议的固定流程

如果把这套方法做成一个 OpenDesign 产品网站设计 skill，我建议它固定执行以下步骤：

1. 收集产品资料
2. 总结目标用户、产品介绍、用途、功能、价格和联系方式
3. 选择或生成产品网站模板
4. 生成适合 OpenDesign 的 prompt
5. 提供参考设计方向，包括布局、字体、动效和价格展示
6. 生成 first draft
7. 检查 Hero 是否吸引人
8. 检查内容是否完整、准确、通顺
9. 检查页面顺序和布局是否合理
10. 检查动效和视觉是否符合 prompt
11. 输出修改建议
12. 整理一份可以给老板看的版本说明

## 建议的 Skill 输入

为了让 skill 稳定运行，输入可以包括：

- 产品名称
- 产品资料
- 目标用户
- 网站目标
- 参考网站
- 品牌或产品调性
- 必须展示的功能
- 价格信息
- 联系方式
- 是否需要动效
- 是否需要多语言
- 老板或客户的特殊偏好

## 建议的 Skill 输出

Skill 最好输出以下内容：

- 网站结构大纲
- 整理后的产品网站文案
- OpenDesign prompt
- 参考设计方向
- first draft 检查清单
- 修改建议
- 给老板看的版本说明

## 可复制的方法论

这套流程里可以标准化的部分包括：

- 资料整理
- 网站结构
- prompt 模板
- 参考维度
- first draft 检查清单
- 修改建议格式

仍然需要人工判断的部分包括：

- 产品理解
- 文案是否准确
- 网站是否符合产品调性
- Hero 是否真正有吸引力
- 老板或客户是否满意

## 结论

我的 OpenDesign 工作流本质上是一个产品网站快速设计流程：

先用模板保证结构，再用 LLM 整理内容和生成 prompt，然后用 OpenDesign 快速出稿，最后通过人工判断和 AI 检查持续迭代。

这套流程的核心优势是速度快、结构清楚、适合反复修改，能够快速产出一个可以给老板评审的 first draft。如果进一步做成 skill，可以把资料整理、prompt 生成、设计检查和版本说明标准化，从而让产品网站设计变得更稳定、更容易复制。
